How to avoid "default to hedging" behaviors
- Seems to happen when an implicit requirement (something the model has reasoned will be needed) has no explicit documentation.  Rather than proactively exploring for a clear yes/no answer or a feasible solution, the model tends to "hedge" which is a strong enough interruption state that it will override Wiggum Loop stop hook criteria and break out of the loop.  Tends to end in a "wait for the user to get this sorted out" state.

Case in point: uncertainty about tmux environment - 
  <★ Insight ─────────────────────────────────────
  Why the hedging occurs: CLAUDE.md (loaded in every session) contains zero mention of the tmux runtime environment. The only tmux reference is a NEVER guardrail about send-keys -l on line 25 — which
  paradoxically implies tmux exists without ever stating it. Meanwhile, dev-session-instructions.md (which IS tmux-aware) only loads for W5:Jarvis-dev sessions. When running as W0:Jarvis, there's no
  documentation affirming "you are in tmux." Without an explicit environment declaration, the model defaults to uncertainty.
  ─────────────────────────────────────────────────

  The fix: add a Runtime Environment section to CLAUDE.md so ALL sessions — not just dev sessions — know about the tmux topology.>
