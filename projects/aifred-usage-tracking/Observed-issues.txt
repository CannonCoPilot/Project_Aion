# 5. Fix Known Pipeline Weaknesses

## 5.1 Reviewer too lenient
Reviewer failed to catch:
- wrong file paths
- failure to read source texts
- weak or hallucinated completion
- structurally incorrect outputs

### Fix
Tighten review criteria without causing infinite review cycling.

### Add checks for
- required source files actually read
- output paths correct
- expected deliverables present
- work grounded in provided inputs
- parent/child task consistency

## 5.2 Stage output path enforcement
Stage service should not emit or approve hallucinated or unvalidated paths.

### Fix
- Validate structured output paths before passing them downstream
- Anchor output paths to project base directory
- Prefer task description authoritative paths over stage-generated guesses
- Prevent writing into repo/project root when task specifies subdirectory targets

## 5.3 Input file provisioning validation
Why did evaluator/reviewer fail to catch missing or unused source files?

### Fix
- Detect whether required inputs were actually accessed
- Reject outputs that claim completion without using mandatory provided inputs

## 5.4 Watcher webhook self-registration
Fix startup registration so watcher does not re-register `localhost` incorrectly.

### Implement
- `WEBHOOK_CALLBACK_HOST` env var
- default to `host.docker.internal` where appropriate
- verify no regression on restart

## 5.5 Watcher log buffering
Operational logs need to flush in real time.

### Implement
- `PYTHONUNBUFFERED=1`
- separate Flask access logs from pipeline operational logs
- verify heartbeats and stage launches appear immediately

Work is complete when all of the following are true:

- Board no longer mislabels tasks as blocked

- Event-driven transitions are primary, with polling as fallback
- Watcher/webhook configuration is durable across restarts
- Reviewer catches objective path/input/output failures
- Stage output paths are validated and anchored correctly
- Parent/master tasks no longer redundantly re-execute completed child work
- Live telemetry/activity history is visible or clearly scoped/documented
- Pulse API usage is corrected and documented
- Structural validation tests pass on GitHub Actions
- Benchmark task set is simplified to a small, repeatable gospel test
- Session inheritance/fork-resume behavior is verifiable
- Logging supports postmortems and user transparency
- Unneeded shells/monitors/processes are shut down
