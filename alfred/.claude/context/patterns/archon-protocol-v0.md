# Archon Protocol v0 — Specification

**Status**: v0, active
**Date**: 2026-04-23
**Companions**: `.claude/context/decisions/archon-architecture.md` (direction)
**Author**: Pulse P2-T01

> **Read first**: Archon v0 is **task-emission only**. Archons write tasks to a
> Pulse board. They do NOT receive commands from AIFred. See `archon-architecture.md`
> for the rationale and the v0/v1 split. This document is the implementation
> contract — everything a reference utility (`archon/register`,
> `archon/task-create`) needs to behave correctly without further design input.

---

## 1. Identity & Registration

### 1.1 Archon identifier

Every Archon has a single globally-unique identifier:

```
archon:<name>
```

Where `<name>` matches `[a-z][a-z0-9_-]{1,63}` (lowercase ASCII, optional
digits/underscores/hyphens, max 64 chars). Examples:

- `archon:jarvis`
- `archon:jarvis-dev`
- `archon:headless-dispatcher`
- `archon:mtg-card-sales`
- `archon:familysearch-hw`

**Suffix convention**: `-dev` denotes the sibling development instance of a
prod Archon (e.g. `archon:jarvis` vs `archon:jarvis-dev`). Dev archons emit to
a separate Pulse board (see P0-T03 decision).

### 1.2 Registration manifest

Optional but recommended: an Archon announces itself once on first boot (and
again if its capabilities change) by POSTing a registration task:

```json
POST /api/v1/tasks
{
  "id": "<auto-generated AION-xxx>",
  "title": "[archon:<name>] Registration",
  "description": "<free-form — markdown OK>",
  "priority": "medium",
  "labels": [
    "archon:<name>",
    "archon:registration",
    "source:<source-identifier>",
    "capability:<primary>",
    "domain:<primary>",
    "risk:safe",
    "archon-version:v0"
  ],
  "metadata": {
    "archon": "<name>",
    "version": "<semver-or-build-id>",
    "host": "<hostname>",
    "manifest": {
      "capabilities": ["..."],
      "domains": ["..."],
      "emits_kinds": ["..."]
    },
    "registered_at": "<ISO-8601 UTC>"
  }
}
```

The registration task may be **closed immediately** by the same Archon with
`reason: "auto-registration complete"`. Its sole purpose is to leave an
audit trail (one event in the `events` table per registration) so humans can
query "what Archons are currently active" by label.

No central registry service exists in v0. A consumer lists Archons by:

```
pulse list --label archon:registration --status closed
```

…and selecting the most recent per `archon:<name>`.

### 1.3 Liveness

An Archon is **active** if it has emitted *any* task within the last 24 hours.
Otherwise:

- `idle` (last emission 1–24h ago)
- `stale` (> 24h)

Liveness is a derived view, not a label stored on the Archon. Compute it from
Pulse events. No automatic restart/eviction is performed in v0.

## 2. Task Lifecycle Contract

### 2.1 The only side of the protocol that exists in v0

Archons are producers. Pulse is the broker. AIFred-Pro is one of several
consumers. Consumers are out of scope for this document.

```
  Archon ─ POST /api/v1/tasks ─▶ Pulse ─── events ───▶ (Dashboard, Executors, Humans)
```

### 2.2 Create contract

Minimum-viable create call:

```json
POST /api/v1/tasks
{
  "title": "<imperative summary, ≤100 chars>",
  "description": "<free-form body, markdown OK>",
  "priority": "low|medium|high|urgent",
  "labels": [
    "archon:<name>",                    // REQUIRED — identifies the producer
    "source:<source-identifier>",       // REQUIRED — what inside the archon emitted this
    "capability:<capability>",          // REQUIRED
    "domain:<domain>",                  // RECOMMENDED
    "risk:<level>",                     // RECOMMENDED
    // …any other labels the Archon wants to apply
  ],
  "metadata": { /* free-form, JSON-serializable */ }
}
```

The `id` field is optional on create; if omitted, Pulse assigns `AION-<hex8>`.

Pulse returns `{"id": "AION-xxxxxxxx", "title": "...", "status": "open",
"labels": [...]}`.

### 2.3 What an Archon may do to its own tasks

| Operation | Allowed | Notes |
|-----------|:-------:|-------|
| Create | ✅ | Primary purpose |
| Add label | ✅ | Via `POST /api/v1/tasks/{id}/labels` or `pulse label add` |
| Remove label | ✅ | Only labels the Archon previously added; best effort |
| Append notes | ✅ | Via `PATCH` with `append_notes` |
| Update description | ⚠️ | Only if correcting / enriching — never destructively |
| Update status | ❌ | **Archons do NOT transition tasks.** That's a consumer's job. |
| Close | ❌ | Archons do NOT close tasks. Consumers do. |
| Delete | ❌ | Not supported in v0. |

### 2.4 Claim by consumers

Consumers (AIFred task-executor, humans, other tooling) claim tasks via:

```
PATCH /api/v1/tasks/{id}
{"status": "in_progress", "claim": true, "actor": "<consumer-id>"}
```

Archons MUST NOT claim. The `claim` flag writes the `claimed_by` column so
that contention between consumers is visible.

## 3. Capability Declaration

An Archon declares what it can emit by attaching `capability:<name>` labels
to every task it creates. The v0 closed set is:

| Capability | Meaning | Examples |
|------------|---------|----------|
| `capability:infra` | Host / OS / Docker / filesystem ops | "rotate logs", "docker restart" |
| `capability:code` | Source code changes | "fix bug", "rename symbol" |
| `capability:docs` | Documentation only | "update CHANGELOG" |
| `capability:decision` | Requires human judgment call | "choose persistence backend" |
| `capability:review` | Peer/self review of existing artifact | "review PR #42" |
| `capability:research` | Exploratory investigation | "survey options for X" |
| `capability:security` | Credentials, auth, audit | "rotate token" |
| `capability:testing` | Test authoring or execution | "add regression test" |
| `capability:protocol` | Changes to the Archon contract itself | Rare |

Additive extensions MUST be prefixed (`capability:x-<name>`) and documented in
the Archon's registration manifest `capabilities` field.

### Registration manifest → capability labels

Every `capability:*` an Archon emits MUST appear in its registration
manifest's `capabilities` array. Consumers may refuse work from an Archon
claiming a capability it didn't register.

## 4. Label & Metadata Conventions

### 4.1 Required labels on every emitted task

| Label | Purpose | Notes |
|-------|---------|-------|
| `archon:<name>` | Producer identity | SSOT for "who wrote this" |
| `source:<source>` | Inside-the-archon origin | e.g. `source:jarvis-session-end`, `source:plan-2026-04-22` |
| `capability:<cap>` | Work kind | From §3 |

### 4.2 Recommended labels

| Label | Purpose |
|-------|---------|
| `domain:<area>` | Subject-matter classification (`domain:security`, `domain:git`, `domain:documentation`) |
| `risk:<level>` | `risk:safe`, `risk:moderate`, `risk:destructive` |
| `project:<name>` | Ties the task to a registered project |
| `target:<env>` | `target:aifred-pro`, `target:aifred-pro-dev`, etc. — tells consumers where to act |
| `phase:<n>` / `plan-id:<pid>` | Plan-tracking identifiers when the emission is part of a numbered plan |
| `auto:candidate` / `auto:ready` | Automation-eligibility hints (see §4.4) |
| `type:<kind>` | Orthogonal classifier (`type:parent`, `type:research`, `type:idea`) |

### 4.3 Reserved / special labels consumers set (Archons MUST NOT set these)

| Label | Set by |
|-------|--------|
| `pipeline:approved` | Human reviewer |
| `pipeline:needs-approval` | Scoring job |
| `completed-by:<actor>` | Executor at close time |
| `blocked:dependency` | Plan-unblocker or scoring job |
| `blocked-by:<plan-id>` | Plan authoring tool |
| `stage:*` | Workflow state machine |

An Archon MAY include `stage:queue` on creation as a hint. It MUST NOT
transition through other stages.

### 4.4 Automation routing

`auto:candidate` (default, implied) means "a scoring job should decide".
`auto:ready` means "the emitter asserts this is directly actionable by an
executor". Archons SHOULD NOT emit `auto:ready` unless they are plan-authoring
tools that already know dependency graph.

### 4.5 Metadata schema

The `metadata` object is free-form JSON. Recommended keys:

```json
{
  "archon": "<name>",             // duplicate of archon:<name> label for convenience
  "emitted_at": "<ISO-8601>",
  "correlation_id": "<opaque>",   // groups related emissions
  "source_ref": "<url-or-path>",  // pointer back to the thing that caused the emission
  "severity": "info|warn|critical"
}
```

Consumers MUST tolerate unknown keys.

## 5. Failure Modes

### 5.1 Pulse unreachable

Archons SHOULD retry with exponential backoff (base 1s, max 60s, up to 5
attempts). On terminal failure:

1. Write the would-be task to a local spool file (`~/.cache/archon-<name>/spool.jsonl` or equivalent).
2. Drain the spool at next successful connect.
3. Log at severity `warn`.

The v0 protocol does **not** specify a spool format beyond "JSONL where each
line is the create-task body". Implementations choose.

### 5.2 Duplicate emission

Pulse has no natural dedup key. If an Archon may re-emit the same task on
retry, it SHOULD include a `correlation_id` in metadata and check whether a
prior open task with that id exists (GET `/api/v1/tasks?label=correlation-id:<id>`) 
before creating. No server-side enforcement.

### 5.3 Invalid label

Pulse currently accepts any label string. Archons SHOULD validate labels
against the conventions above before emission so the board stays queryable.

### 5.4 Capability drift

If an Archon emits with a capability it did not register, behavior is defined
as: the task is stored, dashboard shows it, but **consumers may ignore it**
(no guaranteed action). Archons SHOULD re-register after changing capabilities.

### 5.5 Manifest staleness

If an Archon goes `stale` (§1.3) and then re-emits with changes, it SHOULD
post a fresh registration task before the first work task so its capabilities
manifest is current.

## 6. v0 Scope Boundaries

Explicitly **IN v0**:

- Task creation with labels + metadata
- Optional registration tasks
- Label add/remove/append-notes on Archon-owned tasks
- Capability declaration by label + manifest
- Local spool retry under network failure

Explicitly **OUT OF v0** (deferred to v1):

- Bidirectional command channel (AIFred → Archon)
- Authenticated Archon sessions (tokens, mTLS)
- Rate limiting / per-Archon quotas
- Inter-Archon messaging
- Heartbeat-based eviction
- Server-side deduplication
- Capability negotiation / handshake beyond label inspection
- Archon-level billing / cost attribution

Trust model in v0: **network-based**. Any process that can reach the Pulse
API on the local/shared network can emit as any Archon. The `archon:<name>`
label is an honor-system attribution. This is acceptable because:

1. All current deployments are on a single trusted host (JARVIS.local).
2. `source:` and `metadata.source_ref` let consumers triangulate provenance.
3. Pulse's `events` table provides an immutable audit trail.

## Reference implementations

- `archon/register` (P2-T03) — command-line helper that reads a manifest
  YAML and posts §1.2 registration. Built in AIFred-Pro-Dev, promoted
  via P2-T03p.

  requirement in this document against a live Pulse. Built in
  AIFred-Pro-Dev, promoted via P2-T05p.

P2-T05 suite. If you find one without, open a Pulse task labelled
`archon:registration` + `type:protocol-gap`.
