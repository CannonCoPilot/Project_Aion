## 2. P0 FINDING — UPS Hook Stdin Lacks `context_window`

**Discovered during this audit**: roadmap §4.1 specifies UserPromptSubmit hook reading `context_window` from stdin. **This field is not present in any hook event's stdin payload.**

### Evidence (triangulated)

1. **Official documentation** (https://code.claude.com/docs/en/hooks.md):
   > "Common Input Fields (All Hooks): session_id, transcript_path, cwd, permission_mode, hook_event_name"

   No hook event documents `context_window` in its input schema.

2. **Empirical**: `.jicm-stdin-debug.json` from Jarvis-Dev session captures a real UPS payload:
   ```json
   {"session_id":"...","transcript_path":"...","cwd":"...",
    "permission_mode":"bypassPermissions","hook_event_name":"UserPromptSubmit",
    "prompt":"what is 2+2"}
   ```
   No `context_window`, no `model`, no `cost`, no `rate_limits`.

3. **Prototype self-evidence**: `.jicm-state-hook.json` (Jarvis-Dev) shows `tokens: 0`, `model_id: "unknown"` — the prototype's defensive `// 0` jq fallbacks fire because the data is missing.

4. **Source-citation error**: roadmap §4.1 cites `src/components/StatusLine.tsx:90` and `src/tools/AgentTool/built-in/statuslineSetup.ts:54`. These references describe **statusLine command** payload, NOT UserPromptSubmit hook payload. The plan author conflated the two contracts.

## 3. ARCHITECTURAL CORRECTION — JSONL Transcript Parsing

**New canonical sensing source**: parse the JSONL transcript at `transcript_path` (which IS in every hook stdin) and extract the latest assistant message's `usage` object.

### Schema

Each assistant message in the JSONL transcript carries:
```json
{
  "type": "assistant",
  "message": {
    "usage": {
      "input_tokens": <int>,
      "cache_creation_input_tokens": <int>,
      "cache_read_input_tokens": <int>,
      "output_tokens": <int>,
      "cache_creation": {"ephemeral_5m_input_tokens": <int>, "ephemeral_1h_input_tokens": <int>}
    }
  }
}
```

### Canonical formula

```
current_context_tokens = usage.input_tokens
                       + usage.cache_read_input_tokens
                       + usage.cache_creation_input_tokens
```

### Verification — 2026-05-02 ~18:00Z

| Source | Tokens | Method |
|---|---|---|
| **JSONL latest assistant `usage`** | **220,665** | `1 + 215,466 + 5,198` |
| JSONL second-to-last | 223,203 | `6 + 12,875 + 210,322` |
| **v7 capture-pane** | **228,637** | TUI status line parse (`.jicm-state`) |
| Δ | +7,972 (+3.6%) | structurally explained — see below |

### Why the Δ?

JSONL `usage` is written when each API request COMPLETES — describes what was JUST SENT (past-tense).
TUI status line shows what WILL BE SENT on the next request including the unsubmitted user prompt + system reminders (future-tense).

**JSONL is a strict CONSERVATIVE LOWER BOUND** for "context about to be sent". For threshold triggering, this means:
- If JSONL says 220K and threshold is 300K, we still have ≥80K headroom
- Any over-trigger from JSONL is an under-estimate that fires slightly EARLY — safety-positive
- Eliminates ANY dependence on TUI parsing or statusLine cadence

**Approval**: User approved this correction this session (2026-05-02). Conservative bias is acceptable.

All four observably function — Stop hook firing is empirically verified.

### Stop hook stdin extension

Per docs (`code.claude.com/docs/en/hooks-guide.md`): Stop hook adds `stop_hook_active: bool` to common fields. The new `jicm-stop.sh` will check this and short-circuit if `true` (avoid recursive halts when JICM itself triggers a stop chain).

### `jicm-stop.sh` design (Phase 7.9.1)

```bash
#!/bin/bash
INPUT="$(cat)"
[[ "$(echo "$INPUT" | jq -r '.stop_hook_active // false')" == "true" ]] && exit 0

# Read pending action from state file (written by jicm-gate.sh)
STATE_FILE="$PROJECT_DIR/.claude/context/.jicm-state-hook.json"
PENDING=$(jq -r '.pending_action // ""' "$STATE_FILE" 2>/dev/null)
[[ "$PENDING" != "HALT_AFTER_RESPONSE" ]] && exit 0

# Write the actuator-trigger signal
SIGNAL="$PROJECT_DIR/.claude/context/.jicm-clear-now.signal"
TOKENS=$(jq -r '.tokens // 0' "$STATE_FILE")
THRESHOLD_TYPE=$(jq -r '.action // "unknown"' "$STATE_FILE")
TS=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
echo "{\"threshold_type\":\"$THRESHOLD_TYPE\",\"tokens\":$TOKENS,\"ts\":\"$TS\"}" > "$SIGNAL"

# Clear pending_action atomically (Phase 7.9.1 task #1: jicm-state-update.sh)
"$PROJECT_DIR/.claude/scripts/jicm-state-update.sh" --clear-pending

exit 0
```

## 6. Encoding Directive (User, 2026-05-02)

> Prefer parameters that report in **exact token counts** as the source of truth over parameters that report in **usage %**.

### Implementation rules

| Concept | Encoding |
|---|---|
| Threshold values | Token integers: `JICM_HARD_TOKENS=650000` (NOT `JICM_HARD_PCT=65`) |
| State file primary fields | `tokens`, `cache_read_tokens`, `cache_creation_tokens`, `soft_threshold_tokens`, `hard_threshold_tokens` |
| State file derived fields | `used_percentage` (computed: tokens/window×100; for display only) |
| Window size | From a model→window lookup table; fallback to 1,000,000 (Jarvis runs opus-4-7[1m]) |
| Burn rate | `burn_rate_tpm` (tokens per minute, integer) |
| ETAs | `soft_eta_min`, `hard_eta_min` (computed from burn rate; integer minutes) |

### Window-size lookup (proposed)

```
opus-4-7[1m]  → 1,000,000
opus-4-7      →   200,000
sonnet-4-6    →   200,000
haiku-4-5     →   200,000
unknown       → 1,000,000  # safe upper bound for opus 1M default
```
