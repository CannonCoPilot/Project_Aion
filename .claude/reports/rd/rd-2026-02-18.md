## Research Topics

### 1. Stop Hook `stop_hook_active` Field Semantics (ADOPT)

**Source**: [Claude Code Hooks Reference](https://docs.claude.com/en/docs/claude-code/hooks)

**Finding**: The `stop_hook_active` boolean in Stop hook input has specific semantics:
- `false` → First-time stop, hook hasn't blocked yet → safe to evaluate
- `true` → A Stop hook already blocked previously → should allow through to prevent infinite loops

**Official pattern**:
```bash
if [ "$(echo "$INPUT" | jq -r '.stop_hook_active')" = "true" ]; then
  exit 0  # Allow stop — we already had our chance
fi
```

**Impact**: Our exit-guard currently uses the `.exit-ceremony-done` signal file for loop prevention (Layer 3d). The official `stop_hook_active` check is cleaner and doesn't require filesystem state. However, our signal file approach works across multiple Stop hooks (exit-guard + idle-hands + stop-hook), so both mechanisms have value.

**Recommendation**: ADOPT — Add `stop_hook_active` check as Layer 0 in exit-guard.sh for defense-in-depth. Keep `.exit-ceremony-done` as backup for cross-hook coordination.

### 2. Async Hook Support (`async: true`) (ADAPT)

**Finding**: Claude Code now supports `async: true` on hooks (released January 2026). Async hooks run in the background without blocking Claude's execution.

```json
{
  "type": "command",
  "command": "node backup-script.js",
  "async": true,
  "timeout": 30
}
```

**Candidates for async conversion**:
- `usage-tracker.js` (PostToolUse) — logging only, no blocking needed
- `observation-tracker.js` (PostToolUse) — logging only
- `cross-project-commit-tracker.js` (PostToolUse) — logging only
- `session-tracker.js` (Notification) — logging only

**Impact**: Could reduce hook execution overhead on every tool call. Currently 8+ hooks fire on various events; making loggers async would speed up the pipeline.

**Recommendation**: ADAPT — Convert pure-logging hooks to async. Keep blocking hooks (safety guards, permission gates) synchronous.

### 3. Prompt-Based Stop Hooks (DEFER)

**Finding**: Claude Code supports `type: "prompt"` for Stop hooks, where an LLM evaluates whether to block. This could theoretically replace our bash-based exit-guard with an LLM that reads the transcript directly.

**Recommendation**: DEFER — Our bash-based approach is faster (0ms LLM cost) and now correct. Prompt-based hooks would add latency and cost to every turn end.
